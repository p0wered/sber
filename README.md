## Как развернуть шаблон локально?

### Установить нужную версию Node.js:

Для корректной работы проекта необходимо иметь установленной одну из
нижеперечисленных версий Node.js:

* v18.18.0 и выше
* v20.9.0 и выше
* v21 и выше

### Клонировать проект
```shell
ssh://git@git.infg.tech:10022/templates/s-denga-template.git
```

### Установить модули
```shell
npm install
```

### Запустить локально
```shell
npm run dev
```

## ESLint и Code Style
В проекте используется ESLint для поддержания чистоты и единообразности кода.
Для того чтобы ESLint корректно работал внутри IDE необходимо сделать следующее:

### **В PhpStorm/WebStorm:**
1. Перейти в настройках по пути Settings → Languages & Frameworks → JavaScript → Code Quality Tools → ESLint
2. Выбрать опцию Automatic ESLint Configuration
3. Перезагрузить IDE

### **В VS Code/Cursor:**
1. Скачать плагины ESLint _(dbaeumer.vscode-eslint)_ и Vue.js _(Vue.volar)_
2. Перезагрузить IDE

Правила с комментариями находятся в **[eslint.config.ts](eslint.config.ts)**

### Стиль написания имен файлов:
* Vue компоненты - **PascalCase**
* Ассеты (_svg, png, webp и т.д._) - **kebab-case**
* Файлы TypeScript, JavaScript, JSON и т.д. - **kebab-case**

### Стиль написания кода:
Код должен проходить по правилами линтеров без ошибок, иначе команда `npm run build` и попытки
залить изменения в гит будут отвергаться

Код внутри тега script во всех файлах имеет один отступ. Делать это желательно, но необязательно,
так как это моё личное предпочтение -_-

Также, импорты в проекте располагаются в особом порядке. Сначала переменные внутри { }, затем любые
переменные, и только затем прочее (Vue компоненты, файлы стилей, медиа и т.д.). Это правило тоже
носит рекомендательный характер

## Основная информация

### Конфиг
Конфиг с данными сайта для dev подгружается локально из **[config.local.json](src/config.local.json)**.
Для prod версии конфиг берётся через API запрос на `api/legal-data`

Внутри template к конфигу обращаться через глобальную переменную **$config**. Например:
```html
<span>
  Цена подписки - {{ $config.sum }} рублей
</span>
```

Внутри script через переменную config, импортируемую из **[config.ts](src/config.ts)**. Например:
```typescript
import { config } from "@/config";

const apiUrl = `api/example-api?sid=${config.siteId}}&cid=${config.cid}`;
```

При каких-либо изменениях в prod конфиге, их нужно внести в локальный файл конфига, а также в типизацию
конфига в [global-types.d.ts](src/global-types.d.ts)

### Версии шаблона

Глобально существует две версии каждого шаблона - публичная версия и приватная версия. То, какая версия отображается,
определяется query параметром **utm_source** в URL (который потом записывается в cookie).

В публичной версии отображается весь контента сайта, включая полную анкету, информацию о подписке, все popup окна и т.д.
Публичная версия существует для прохода всех необходимых проверок от модераторов и банков.

В приватной версии весь _лишний_ контент скрыт от глаз пользователя, а анкета оформления заявки изменена _(подробнее
описано ниже)_.

### Анкета

Основная часть сайта - это анкета оформления заявки. Всего она состоит из 5 шагов:

1. **Контактные данные** (Номер телефона, Email)
2. **Личные данные** (ФИО, Дата рождения, Пол)
3. **Паспортные данные** (Номер и серия паспорта, Код подразделения, Дата выдачи паспорта)
4. **Верификация** (Показ документов, подписание их через номер телефона)
5. **Шаг оплаты** (Привязка карты пользователя)

На первых четырёх шагах заполненные данные отправляются на бэк через API запросы `https://api.mfo-0.ru/new-user` и
`https://api.mfo-0.ru/add`. Первый запрос создаёт пользователя в БД, id которого является lead_id, и записывает к нему
переданные данные (любые из списка, в любом порядке). Если пользователь с таким lead_id уже существует, то к нему просто
записываются передаваемые данные. По факту эти два запроса ничем не отличаются на бэке, но зачем-то разделены на два
разных роута.

Далее, существуют два вида отображения анкеты:

1. **Полная версия анкеты**. Отображается на публичной версии сайта. Состоит из всех 5 шагов, номер телефона
   отправляется на бэк, только после подтверждения с его помощью документов.
2. **Короткая анкета**. Отображается на приватной версии сайта. Состоит из двух шагов - Контактная информация,
   Форма оплаты. На шаге верификации отсутствует подтверждение документов.

После прохождения анкеты и успешной привязки карты, пользователя перекидывает на страницу витрины с предложениями
[Companys.vue](src/views/Companys.vue) с query параметрами `activation=1` и `uid={lead_id пользователя}`. При монтировании
этой страницы с query параметром `activation=1` происходит отправка постбэка через API запрос
`https://api.mfo-0.ru/set-verify?site_id=${config.siteId}`, где siteId - id сайта из конфига (подробнее про витрины в
разделе ниже).

Основной файл анкеты - [Anketa.vue](src/views/Anketa.vue) объединяет в себе все элементы анкеты и управление ими.
Элементы анкеты представляют собой отдельные Vue компоненты, используемые внутри Anketa.vue, они находятся в папке
[Anketa](src/components/Anketa). Для управления данными анкеты в localStorage используются функции из
[anketa.ts](src/utils/anketa.ts) _(подробнее в разделе ниже)_.

### Постоянное хранение данных

Все данные, связанные с анкетой, хранятся внутри **localStorage**. Это нужно, чтобы при повторном заходе на сайт,
данные заявки пользователя были уже заполнены. Названия и формат данных одинаковы на всех шаблонах. Это обязательно,
так как на одном домене могут существовать несколько шаблонов.

Данные о сумме и сроке займа хранятся внутри sessionStorage в определённом формате (так исторически сложилось).

Различные метки, в том числе и utm_source, хранятся в **cookie** хранилище.

Также, в шаблонах присутствуют запись подобных данных из query параметров URL в различные хранилища. После их успещной
записи эти данные удаляются из URL. Но стоит учитывать, что это происходит только на определённых страницах, так как на
некоторых других страницах шаблона query параметры необходимы для правильной работы их логики.

Для лучшего понимания советую ознакомиться с кодом функций `saveParams`, `removeQuery` и `useUtmSource` в файле
[common.ts](src/utils/common.ts), а так же с тем, как они используются в проекте.

### Витрины

Помимо анкеты, важной частью шаблонов являются витрины с предложениями партнёров. Они находятся по пути
/apply/companys (да, именно с грамматической ошибкой, так тоже исторически сложилось). То, какой список предложений
отображается, определяется query параметрами. Т.е. в зависимости от query параметров выбирается id витрины,
запрашиваемой с бэка:

```typescript
const offersId = {
  default: 95,
  double: 96,
  comebacker: 97,
  overloaded: 98
} as const;
```

На каждом шаблоне свой набор id

Всего в /apply/companys четыре вида витрин:

1. **Основная**. Дефолтная, без query параметров.
2. **Double**. Открывается незаметно для пользователя при начале оформления заявки. Имеет ?type=double в ссылке.
3. **Comebacker**. Открывается при попытках пользователя вернуться назад с внутренних страниц шаблона. Имеет  
   ?type=comebacker в ссылке.
4. **Overloaded**. Открывается при долгом бездействии на сайте. Имеет ?type=overloaded в ссылке.

### API
Для API запросов в проекте используется модуль axios, для написания новых запросов использовать его. Многие
запросы в проекте не проходят локально, поэтому их придётся видоизменять для отладки.

Для отладки API запросов внутри анкеты имеется функция **simulateApiRequest** в **[anketa.ts](src/utils/anketa.ts)**, на
которую нужно заменить все апи запросы в **[Anketa.vue](src/views/Anketa.vue)** в функции **handleStepSubmit**:
```typescript
const stepHandlers = {
  1: sendStepOneRequest,
  2: sendStepTwoRequest,
  3: sendStepThreeRequest
};
```
Заменить на:
```typescript
const stepHandlers = {
  1: simulateApiRequest,
  2: simulateApiRequest,
  3: simulateApiRequest
};
```

API запросы, которые отправляются сами на себя, тоже не проходят, поэтому для них нужно указать конкретный
домен. Например, API запрос `/api/suggest` в компоненте [InputEmail.vue](src/components/Common/Inputs/InputEmail.vue)
необходимо поменять на `https://maxcash.ru/api/suggest`, чтобы он заработал.
